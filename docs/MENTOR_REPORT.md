# 项目汇报摘要

## 30 秒介绍

我实现了一套 Linux 主机运行时安全原型：Falco 负责发现大范围可疑行为，
BPF LSM 负责在内核中审计或阻断指定敏感操作，TSA 将运行时事件与 Lynis
基线结果融合、去重并评分，最后通过 Web 看板展示完整证据链。目前检测、
内核审计、评分、持久化和可视化流程已经贯通，强制阻断仍处于灰度验证前的
`audit` 阶段。

## 项目目标

本项目面向 Linux 主机运行时安全，解决三个问题：

1. 系统正在发生什么可疑行为；
2. 指定敏感对象是否需要在内核中阻断；
3. 多种安全结果如何统一记录、评分和展示。

## 核心方案

系统由四部分组成：

- **Falco**：广域检测系统调用，发现文件、进程、网络和容器异常；
- **BPF LSM**：在内核安全检查点匹配指定文件，执行审计或拒绝；
- **TSA**：融合 Falco、BPF LSM 和 Lynis，完成去重、限速、持久化和评分；
- **Lynis**：提供补丁、认证、SSH、防火墙等静态基线结果。

```text
运行时操作
   ├─ Falco：检测并报警
   └─ BPF LSM：审计或内核阻断
             ↓
       TSA 融合与评分 ← Lynis 基线
             ↓
       SQLite、JSON、Web 看板
```

## 关键区别

- Falco 报警不等于操作被拦截；
- BPF LSM 的 `audit` 表示记录后放行；
- BPF LSM 的 `enforce` 才表示内核返回 `-EPERM`；
- TSA 不拥有 root 权限，也不修改内核策略，只负责证据融合。

这种分层避免了“检测程序直接控制内核”的高风险耦合。

## 已完成工作

- 主机版 Falco modern eBPF 部署及自定义规则；
- 基于 device/inode 的 BPF LSM Map 策略；
- 写入、删除、重命名、属性修改四类内核检查；
- `audit` 与 `enforce` 两种策略模式；
- BPF ring buffer 结构化事件输出；
- TSA 双事件源融合、去重、限速、风险过期和重启恢复；
- Lynis 基线分与运行时分加权；
- 只读 Web 看板及跨来源事件证据链；
- systemd、日志轮转和一键部署；
- Go 测试及 14 项 Python 单元/集成测试。

## 演示结果

对 `/etc/tmp_rzh` 执行写入时：

1. Falco 产生 `Write below etc` 报警；
2. BPF LSM 命中 `protect_tmp_rzh`；
3. 当前 `audit` 模式产生结构化事件并允许写入；
4. TSA 接收事件并更新临时风险；
5. 看板展示从操作到评分的完整证据链。

这证明了 Falco 检测、BPF LSM 内核决策、TSA 融合和可视化链路已经打通。

## 当前边界

1. BPF 策略当前绑定文件的 device/inode。文件删除后以新 inode 重建，需要重载策略；
2. 当前处于 `audit` 阶段，尚未对生产敏感文件启用强制阻断；
3. Falco 与 BPF 事件关联使用 PID，缺少 PID 时使用“进程名、保护路径和时间窗口”，属于工程关联而非内核级唯一事务 ID；
4. TSA 当前采用规则权重评分，不等同于正式风险量化标准。

## 下一步

优先级从高到低：

1. 增加父目录与文件名策略，解决 inode 重建后的保护连续性；
2. 选取低风险对象开展 `enforce` 灰度与回滚测试；
3. 为跨来源事件增加统一关联 ID；
4. 根据实验数据校准规则权重、白名单和风险有效期；
5. 增加性能测试，量化 BPF LSM 对系统调用延迟的影响。

## 一句话总结

本项目不是简单堆叠多个组件，而是建立了“广域检测、内核决策、风险融合、
可视化证据”四层分工明确的 Linux 主机安全框架。
